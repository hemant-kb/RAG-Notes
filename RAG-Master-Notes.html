Introduction to RAG

What Retrieval-Augmented Generation is, the four problems it solves, and the end-to-end pipeline.

Chapter 0.1

What & Why RAG?

Concept

Retrieval-Augmented Generation (RAG) is an architecture that gives a Large Language Model access to an external knowledge source at query time. Instead of relying only on what it memorised during training, the model first retrieves relevant documents and then generates an answer grounded in those documents.

Answer = LLM( Query + Retrieved_Context )
The retrieved context is injected into the prompt — the model "reads" it before answering.
Intuition

A plain LLM is a closed-book exam: it answers purely from memory and will confidently guess when it doesn't know. RAG turns it into an open-book exam — the model looks up the relevant pages first, then writes the answer.

🏫
Real-world analogy

A brilliant lawyer (the LLM) with a vast but slightly outdated memory. Before answering a client, they walk to the law library (the vector store), pull the exact statutes relevant to the case (retrieval), and argue from those documents (generation) — instead of reciting half-remembered law.

Why it's needed

LLMs are powerful but fundamentally limited by their training data and a fixed context. RAG decouples knowledge from reasoning: the model keeps its language/reasoning skill, while the knowledge lives in an external store you control, update, and audit.

Chapter 0.2

The Four Problems RAG Solves

RAG addresses the four core failure modes of a standalone LLM.
ProblemStandalone LLMWith RAG
Knowledge cutoffStuck at training dateInject up-to-date docs at query time
HallucinationInvents confident, wrong factsGrounded in retrieved evidence
AttributionCannot cite a sourceReturns the exact source chunks
Private dataNever trained on your dataReads your corpus without retraining
✅
Why not just fine-tune?

Fine-tuning bakes knowledge into weights — expensive, slow to update, and still hallucinates. RAG updates knowledge by simply adding a document to the store. Fine-tuning teaches behaviour/style; RAG supplies facts. They're complementary.

Chapter 0.3

The Complete RAG Pipeline

Two phases

Every RAG system has an offline indexing phase (done once, ahead of time) and an online query phase (done per user request).

Indexing happens once (left). Every query flows through embed → retrieve → augment → generate (right), converging on the LLM.
Algorithm
RAG in 3 online steps
  • 1 · Retrieve — embed the query, find the top-k most similar chunks in the vector store.
  • 2 · Augment — concatenate the retrieved chunks into the prompt as context.
  • 3 · Generate — the LLM answers using the augmented prompt, optionally citing sources.
Summary — RAG = retrieve relevant context, then generate a grounded answer. It fixes knowledge cutoff, hallucination, attribution, and private-data access by keeping knowledge in an external, updatable store. The rest of this handbook unpacks every stage of the pipeline above — from how text becomes vectors, to how those vectors are indexed and retrieved, to advanced and production-grade variations.

Evolution of Search Systems

Concept

Information Retrieval (IR) is the science of finding relevant documents for a query. It evolved through four major eras — each fixing the previous one's blind spot.

Each era added a capability: relevance weighting (TF-IDF), better calibration (BM25), and finally meaning beyond exact words (dense retrieval).

Sparse methods (keyword → BM25) match words; dense retrieval matches meaning. Modern RAG often uses both (hybrid search, Part 6) — sparse nails exact terms like part numbers, dense catches paraphrases.

Chapter 2

Vector Space Model & Term-Document Matrix

Concept

The Vector Space Model (VSM) represents every document and query as a vector in a high-dimensional space where each dimension is a vocabulary term. Similarity between a query and a document becomes the geometric closeness of their vectors.

Term-Document Matrix

Lay all documents out as rows (or columns) and all vocabulary terms as the other axis. Each cell holds a weight (count, TF-IDF, etc.) of that term in that document.

Term \ DocD1D2D3

retrieval100
training031
overfitting020
Each column is a document vector. A query becomes a vector too — retrieval ranks docs by cosine similarity to it.

Cosine measures the angle between two vectors, ignoring length:

cos(Q, D) = (Q · D) / ( ‖Q‖ · ‖D‖ )
Significance: the dot product Q·D counts shared terms (weighted); dividing by the lengths ‖Q‖‖D‖ normalizes away document size, so a long doc can't win just by being long.
‖Q‖ = √(1²+1²) = √2 ≈ 1.414

D3 wins — it contains both query words ("cat" and "dog"), while D1 and D2 each contain only one. The geometry captured exactly what we wanted: more shared meaning → smaller angle → higher score.

💡
Why represent text as vectors at all?

Because once text is a vector, relevance becomes geometry. "Find relevant documents" turns into "find the nearest vectors" — a problem computers solve extremely fast. Every later idea (TF-IDF weights, dense embeddings, ANN indexes) is just a better way to place these vectors.

Sparse vs Dense representation

Sparse (VSM / TF-IDF / BM25)

  • One dimension per vocabulary word (10k–1M+ dims)
  • Mostly zeros — a doc touches few terms
  • Interpretable, exact-match friendly
  • Can't see "car" ≈ "automobile"

Dense (embeddings)

  • Fixed small size (384–3072 dims)
  • Every dimension is non-zero, learned
  • Captures meaning, synonyms, paraphrase
  • Needs a trained model + vector index
⚠️
Vocabulary mismatch (the core motivation for dense retrieval)

Sparse models fail when the query and document use different words for the same idea: a query "car insurance" won't match a doc saying "automobile coverage". This vocabulary mismatch problem is exactly what dense embeddings solve — covered fully in Part 2 & Chapter 5.

Chapter 3

TF-IDF

Concept

TF-IDF (Term Frequency × Inverse Document Frequency) scores how important a word is to a document relative to the whole collection. Words that appear often in one doc but rarely across the corpus get the highest weight.

Mathematical explanation
tf-idf(t, d) = tf(t, d) × idf(t)
idf(t) = log( N / df(t) )
tf(t,d) = times term t appears in doc d  ·  N = total docs  ·  df(t) = number of docs containing t. Rare term ⇒ small df ⇒ large idf ⇒ high weight.
Intuition
  • TF rewards a word for appearing a lot in this document.
  • IDF punishes a word for appearing in many documents ("the", "is" → near-zero).
  • Product is high only for words that are frequent here but rare elsewhere — the distinctive terms.
Worked example

Corpus of N = 1000 documents. The word "the" appears in all 1000; "transformer" appears in 10.

  • idf("the") = log(1000 / 1000) = 0 → contributes nothing, regardless of how often it appears.
  • idf("transformer") = log(1000 / 10) = log(100) ≈ 4.6 → highly distinctive.

So a doc mentioning "transformer" 3 times scores 3 × 4.6 = 13.8 on that term, while "the" adds 0.

✅ Advantages

  • Simple, fast, no training
  • Strong baseline, interpretable weights
  • Great for exact-term search

⚠️ Limitations

  • No length normalization → long docs inflate TF
  • Linear TF: 100 occurrences = 100× weight (unrealistic)
  • Lexical only — no synonyms/meaning

BM25 (next chapter) fixes the first two limitations directly.

Chapter 4

BM25 — Best Match 25

Concept

BM25 is the gold-standard sparse retrieval algorithm — the default in Elasticsearch/Lucene. It improves TF-IDF with two ideas: term-frequency saturation and document-length normalization.

Mathematical explanation
BM25(q,d) = Σt∈q  IDF(t) · [ tf(t,d)·(k₁+1) ] / [ tf(t,d) + k₁·(1 − b + b·|d|/davg) ]
IDF(t) = log[ (N − nt + 0.5) / (nt + 0.5) ]  ·  k₁ ≈ 1.5 (TF saturation)  ·  b ≈ 0.75 (length normalization)  ·  |d| = doc length, davg = average length.

The two key parameters

k₁ — TF saturation

Controls how fast extra occurrences stop helping. With k₁=1.5, the 1st occurrence matters a lot, the 10th barely adds anything. Prevents one spammed word from dominating.

b — length normalization

b=1 fully penalizes long docs; b=0 ignores length. Default 0.75 means a long document doesn't win just by repeating a term more times.

Worked example — every number computed

We'll rank 3 documents for the query "ML training overfitting" and follow each step end-to-end. Parameters: k₁ = 1.5, b = 0.75.

D1"ML training involves gradient descent and overfitting detection"8
D2"overfitting means the model memorises training data"7
D3"supervised learning trains models on labelled data"7

Average document length:  davg = (8 + 7 + 7) / 3 = 7.33 words.

Step 3 · IDF per term  —  IDF(t) = log[ (N − nt + 0.5) / (nt + 0.5) ] ,  N = 3

IDF("ML")          = ln( (3 − 1 + 0.5) / (1 + 0.5) ) = ln( 2.5 / 1.5 ) = ln(1.667) = +0.511

⚠️
Why "training" gets a negative IDF here

With the classic formula, a term appearing in more than half the corpus (here 2 of 3 docs) gets a negative IDF — it's so common it's treated as noise. Real engines (Lucene/Elastic) clamp this with max(0, …) or use a +1 variant so scores never go negative. We'll keep the raw values to show the mechanics, then note the clamped result.

Step 4 · The TF-saturation term  —  per document

For each doc we precompute the denominator factor  K = k₁·(1 − b + b·|d|/davg) , then the saturated term  S = tf·(k₁+1) / (tf + K). Since every matching term here has tf = 1, S is the same for any matched term within a given doc.

# length-normalization factor K = k1*(1 - b + b*|d|/d_avg),  k1=1.5, b=0.75
D1 (|d|=8):  K = 1.5*(1 − 0.75 + 0.75*8/7.33) = 1.5*(0.25 + 0.8182) = 1.5*1.0682 = 1.602
D2 (|d|=7):  K = 1.5*(0.25 + 0.75*7/7.33)     = 1.5*(0.25 + 0.7159) = 1.5*0.9659 = 1.449
D3 (|d|=7):  K = same as D2                                                       = 1.449

# saturated TF score for a matched term (tf=1):  S = tf*(k1+1)/(tf+K) = 2.5/(1+K)
S(D1) = 2.5 / (1 + 1.602) = 2.5 / 2.602 = 0.961
S(D2) = 2.5 / (1 + 1.449) = 2.5 / 2.449 = 1.021
S(D3) = 2.5 / (1 + 1.449) = 2.5 / 2.449 = 1.021

Note D2/D3 get a slightly higher S than D1 — they're shorter, so length normalization rewards them. This is the b parameter at work.

Step 5 · Sum IDF × S over matched query terms  —  BM25(q,d)

# BM25 = Σ over matched query terms of  IDF(t) * S(doc)
D1 matches ML, training, overfitting:
   = IDF(ML)*S(D1) + IDF(training)*S(D1) + IDF(overfitting)*S(D1)
   = (+0.511)*0.960 + (−0.511)*0.960 + (−0.511)*0.960

   = (−0.511)*1.021 + (−0.511)*1.021 = −0.522 − 0.522         = −1.044

D3 matches nothing                                            = 0.000

Reading the result

Raw ranking: D3 (0) > D1 (−0.49) > D2 (−1.04). That looks odd — until you see why: "training" and "overfitting" are so common (in ⅔ of docs) that their negative IDF penalises the docs containing them, and the one rare, discriminating term — "ML" — only D1 has. With the real-world clamped IDF (negatives → 0), the common terms contribute 0 and only the rare "ML" matters:

# With Lucene-style clamping: IDF(training)=IDF(overfitting)=0, IDF(ML)=+0.511
D1 = 0.511 * S(D1) = 0.511 * 0.960 = 0.491   ← only doc with the rare term "ML"
D2 = 0 + 0          =               0.000
D3 = 0              =               0.000

Final ranking:  D1 > D2 = D3   ✅  (the distinctive term "ML" decides it)
✅
The takeaway from the numbers

BM25's ranking is driven by rare, discriminating terms (high IDF), dampened by repetition (k₁ saturation) and document length (b normalization). A toy 3-doc corpus exaggerates IDF effects; on a realistic corpus of thousands of docs, IDFs are almost always positive and the formula behaves smoothly.

Chapter 5

Inverted Index & Vocabulary Mismatch

Concept

An inverted index is the data structure that makes keyword search fast. Instead of scanning every document for a term, it maps each term → list of documents containing it (the postings list).

The inverted index turns "which docs contain this word?" into an O(1) lookup + list intersection.
Why it's needed

Without it, every query would scan every document — O(N) per term. The inverted index is the backbone of BM25/Lucene/Elasticsearch and lets sparse retrieval stay fast at web scale.

⚠️
The vocabulary mismatch problem

The inverted index only finds exact tokens. A search for "heart attack" won't surface a doc that says "myocardial infarction" — same meaning, different words. Worse, our example above misses nothing only because words match exactly; a query for "ML model" would return nothing even though D1 is about exactly that ("machine learning model"). No amount of BM25 tuning fixes this — it's a fundamental limit of lexical matching.

✅
The fix, in one line

Dense embeddings (Part 2) map "ML" and "machine learning" to nearby vectors, so retrieval matches on meaning — exactly what the inverted index cannot do.

✅
The bridge to Part 2

Dense retrieval solves vocabulary mismatch by mapping text to meaning vectors, so "heart attack" and "myocardial infarction" land close together. That's the entire motivation for embeddings — which we dive into next.

Interview questions
Why is BM25 still used when dense retrieval exists?
BM25 needs no GPU/model, is interpretable, excels at exact-term matches (IDs, codes, names), and is a strong, cheap baseline. Production systems often combine BM25 + dense (hybrid) rather than replace one with the other.
What do k₁ and b control in BM25?
k₁ controls term-frequency saturation (how quickly repeated terms stop adding score); b controls document-length normalization (how much long docs are penalized). Defaults ≈ 1.5 and 0.75.
What is the vocabulary mismatch problem?
When query and relevant document express the same concept with different words, lexical methods (inverted index, BM25) fail to match them. Dense embeddings address it by matching on semantic meaning.
Common mistakes
Assuming BM25 understands synonyms
It doesn't — it's purely lexical. If recall on paraphrases matters, add dense retrieval (hybrid search).
Forgetting to normalize document length
Plain TF-IDF lets long documents dominate. BM25's b parameter fixes this — don't set b=0 unless you truly want length-agnostic scoring.
Part 1 summary — IR evolved from exact keyword matching → TF-IDF (weight rare terms) → BM25 (saturation + length norm., the sparse gold standard), all powered by the inverted index. Their shared ceiling is vocabulary mismatch: they match words, not meaning. That limitation motivates dense embeddings, the subject of Part 2.

Evolution of Embeddings

Concept

An embedding turns text into a vector of numbers that captures meaning. The field evolved from counting words (no meaning) to deep contextual models that understand a word differently depending on its sentence.

From count-based sparse vectors → static dense word vectors → contextual transformer embeddings → sentence-level models built for retrieval.

Static vs contextual — the big leap

Word2Vec gives "bank" a single vector — same for a river bank and a money bank. BERT gives it a different vector in each sentence. RAG retrieval embeddings are built on this contextual, sentence-level lineage (SBERT, OpenAI, BGE, Nomic…).

How is text actually converted to a vector?

This is the question that confuses most people. Let's trace a sentence through a modern (transformer / SBERT-style) embedding model, step by step.

1
Tokenize — split text into sub-word tokens
"RAG retrieves documents"
   → ["RAG", "retrie", "##ves", "documents"]   (sub-word/WordPiece tokens)

Sub-word tokenization means even an unseen word like "retrieves" is built from known pieces — no out-of-vocabulary problem.

2
Map each token to an integer ID (from the model's vocab)
["RAG", "retrie", "##ves", "documents"] → [10243, 8801, 2229, 5169]
3
Embedding lookup — each ID indexes a row in a learned matrix

The model holds an embedding matrix E of shape [vocab_size × d] (e.g. 30000 × 384). Token ID k just grabs row k:

token 10243 → E[10243] = [0.11, -0.04, …, 0.22]   (a 384-d vector)

These per-token vectors are the starting (static) embeddings — same idea as Word2Vec.

4
Contextualize — the transformer mixes tokens via self-attention

Each token vector is updated by attending to all the others, so "bank" next to "river" ends up different from "bank" next to "money". Output: one contextual vector per token.

[t1, t2, t3, t4] → transformer → [h1, h2, h3, h4]   (each 384-d, context-aware)
5
Pool — combine token vectors into ONE sentence vector

For retrieval we need a single vector per chunk. The most common method is mean pooling: average the token vectors element-wise.

v = (1/n) · Σi=1..n hᵢ
Significance: averaging blends every token's contextual meaning into one fixed-size vector, so two sentences with similar meaning (even with different words) land close together. (Some models instead use the special [CLS] token's vector.)
6
Normalize — scale to unit length
v_final = v / ‖v‖     → now cosine similarity = dot product (fast)

The result, e.g. [0.02, -0.13, …, 0.07] (384-d), is the embedding stored in the vector DB.

Where do the numbers in the embedding matrix come from? (training intuition)

The matrix isn't hand-set — it's learned. Word2Vec's insight (the "distributional hypothesis": a word is known by the company it keeps) is to train vectors to predict context:

skip-gram objective:   maximize  Σt Σ−c≤j≤c, j≠0 log P( wt+j | wt )
For each center word wt, push its vector to predict the surrounding words wt+j within a window of size c.
Significance: words appearing in similar contexts ("king"/"queen", "Paris"/"London") get pulled toward similar vectors — which is exactly why king − man + woman ≈ queen works (Chapter 7).

Chapter 7

Embedding Fundamentals

Concept

An embedding maps text into a point in a continuous embedding space. The defining property: semantically similar text lands close together, and directions in the space encode meaning.

Real-world analogy

Think of a map of a country. Cities (words/sentences) that are culturally or geographically related sit near each other. "Distance on the map" ≈ "difference in meaning." Embeddings build that map automatically from data — just in hundreds of dimensions instead of two.

Semantic similarity & vector arithmetic

Because meaning is encoded as geometry, you can do arithmetic on meaning. The famous example:

vec("king") − vec("man") + vec("woman") ≈ vec("queen")
The "king → man" direction encodes royalty minus gender; adding "woman" lands near "queen".
The gender vector (man→woman) is parallel to (king→queen). Meaning lives in directions, which is why analogies work.
Why it's needed for RAG

This geometry is exactly what lets retrieval match a query to relevant chunks even when they share no words — the answer to the vocabulary-mismatch problem from Part 1. The how-close question is answered by distance metrics (Chapter 9), and the how-many-dimensions question by dimensionality (Chapter 8).

Interview questions
What does it mean that embeddings are "semantic"?
Texts with similar meaning map to nearby vectors regardless of exact wording, so similarity is measured by geometric closeness (e.g. cosine), not word overlap.
Why does king − man + woman ≈ queen work?
Embedding spaces encode attributes as consistent directions. The "gender" offset is roughly the same vector everywhere, so subtracting "man" and adding "woman" shifts "king" along that direction to "queen".
Chapter 8 · Embeddings

Dimensionality of Embeddings

What is dimensionality?

Each embedding is a list of floats: [0.23, -0.41, 0.88, …, 0.05]. The length of this list is the dimensionality. Each dimension loosely encodes some latent semantic feature — gender, topic, sentiment, domain, and so on.

Retrieval quality rises steeply, then flattens. Beyond ~1536 dims (OpenAI ada-002 territory) extra dimensions capture only subtle nuance — while cost keeps growing linearly.

3 · Computational speed — the math

Dot product and cosine similarity both require element-wise multiply + sum:

score(q, d) = Σi=1..D  qi · di
This is O(D) per document comparison, and O(n × D) for a brute-force scan over n documents.

Chapter 9

Distance & Similarity Metrics

Concept

Once text is a vector, "relevance" becomes "closeness". A distance metric defines how we measure that closeness. The three you'll use most in RAG are Cosine similarity, Dot product, and Euclidean (L2); Manhattan (L1) appears occasionally.

Mathematical explanation
Cosine(a,b) = (a · b) / (‖a‖ ‖b‖)  ∈ [−1, 1]  — angle only, ignores magnitude
Dot(a,b) = Σ aᵢbᵢ  — angle and magnitude
Euclidean(a,b) = √( Σ (aᵢ − bᵢ)² )  — straight-line distance (L2)
Manhattan(a,b) = Σ |aᵢ − bᵢ|  — grid distance (L1)
💡
The key relationship

If vectors are L2-normalized (length = 1), then dot product = cosine similarity, and Euclidean distance becomes a monotonic function of cosine. OpenAI & most modern embeddings are pre-normalized — so dot product and cosine give identical ranking, and dot is faster.

Cosine cares only about direction (angle); Euclidean cares about absolute position. For text embeddings, direction usually carries the meaning — hence cosine's popularity.
MetricMeasuresRangeBest forWatch out
CosineAngle (direction)[−1, 1]Text similarity (default in RAG)Ignores magnitude
Dot productAngle + magnitude(−∞, ∞)Normalized vecs (fast); recsysLong vectors score higher
Euclidean (L2)Straight-line distance[0, ∞)Clustering (k-means / IVF)Curse of dimensionality
Manhattan (L1)Axis-wise distance[0, ∞)High-dim, robust to outliersRare for embeddings
Worked example 
# a = [1, 0],  b = [0.7, 0.7]  (45° apart)
dot(a,b)       = 1·0.7 + 0·0.7        = 0.70
‖a‖=1, ‖b‖     = √(0.49+0.49) ≈ 0.99
cosine(a,b)    = 0.70 / (1 · 0.99)    = 0.71   # ~cos(45°)
euclidean(a,b) = √((1−0.7)²+(0−0.7)²) = √(0.09+0.49) = 0.76

All three agree on ranking here; they differ most when vector lengths vary a lot.

Interview questions
Why is cosine similarity preferred for text embeddings?
Meaning is encoded in direction, not magnitude (which often reflects text length/frequency). Cosine normalizes away magnitude, comparing pure orientation — so a short and long passage on the same topic still match.
When are dot product and cosine identical?
When both vectors are L2-normalized (unit length). Then ‖a‖=‖b‖=1, so dot = cosine. Many embedding APIs return normalized vectors, letting you use the faster dot product.
Common mistakes
Mixing metrics between indexing and querying
If your index is built for cosine but you query with raw dot on unnormalized vectors, ranking breaks. Keep the metric consistent and normalize when required.
Forgetting to re-normalize after truncating dimensions
Truncating an embedding (e.g. Matryoshka 3072→256) changes its norm; re-normalize before cosine/dot comparisons.

Part 4

Chunking

Documents are too big to embed whole. Chunking decides how text is split — and it quietly makes or breaks retrieval quality.

Chapter 14

Why Chunking?

Concept

Chunking is splitting a large document into smaller passages before embedding them. Each chunk becomes one vector in the store, and retrieval returns whole chunks. The chunk is therefore the fundamental unit of retrieval — its size and boundaries determine what the LLM ever gets to see.

Why it's needed — three forces
Chunking exists to fit model limits, sharpen retrieval, and keep prompts cheap and focused.
Real-world analogy

A textbook isn't one giant blob — it's chapters, sections, and paragraphs. When you need one fact, you flip to the right paragraph, not re-read the whole book. Chunking gives your retriever those paragraph-sized "pages" to flip to.

💡
The precision ↔ context tension (the whole game)

Too small → each chunk is precise but loses surrounding context ("It increases this by 20%" — what increases what?). Too large → rich context but the vector is unfocused, so retrieval misses or returns irrelevant material. Every chunking decision is navigating this trade-off — and the Parent Document Retriever (Ch 30) is one clever escape from it.

When in doubt, use the Recursive splitter — it respects natural boundaries (paragraphs → sentences → words) and is the default for most RAG pipelines.

Chapter 17

Chunk Overlap

Concept

Overlap repeats the last few tokens of one chunk at the start of the next. It prevents a key sentence from being orphaned by a hard cut sitting right in the middle of it.

Visual example
The shared orange span lives in both chunks, so a query about "overfitting detection" matches whichever chunk is retrieved — the idea is never split across an unrecoverable boundary.

✅ Pros

  • Preserves context across boundaries
  • Stops mid-idea splits from losing meaning
  • Improves recall on boundary-spanning facts

⚠️ Cons

  • Duplicates text → more vectors, more storage
  • Same content can be retrieved twice (redundancy)
  • Too much overlap wastes tokens & cost
🧭
Rule of thumb

Set overlap to 10–20% of chunk size (e.g. 50–100 tokens for a 500-token chunk). Enough to bridge sentences, not so much that you store the document twice.

Interview questions
How do you choose chunk size and overlap?
Start around 500 tokens with 10–15% overlap, then tune empirically against a retrieval eval set (recall/precision). Match the chunk to the content's natural unit (paragraph, function, QA pair). There's no universal best — it depends on data and query style.
Why might too-large chunks hurt retrieval?
A large chunk mixes multiple topics, so its single embedding is an average that matches no query sharply (low precision), and it injects irrelevant text into the prompt (more cost, more noise, more hallucination risk).
How do you keep precision of small chunks but context of large ones?
Use a Parent Document Retriever: index small child chunks for precise matching, but return the larger parent chunk to the LLM for context (covered in Chapter 30).
Common mistakes
Splitting by fixed characters and cutting mid-word/sentence
Use a recursive or structure-aware splitter so cuts fall on natural boundaries instead of slicing "over|fitting".
Zero overlap on dense technical text
A fact spanning a chunk boundary becomes unretrievable. Add modest overlap to bridge boundaries.
Ignoring document structure
Chunking a codebase or markdown doc with a plain character splitter destroys functions/sections. Use the structure-aware splitter for that format.

Vector Databases

Where embeddings live. What's stored per record, how a vector store differs from a vector database, and how to safely create/update/delete vectors.

Chapter 18

Vector Store Structure

Concept

A vector store keeps one record per chunk. Each record bundles four things: a unique ID, the embedding vector, the original text, and arbitrary metadata.

The embedding powers similarity search; the text is what's returned to the LLM; the metadata enables filtering, citations, and access control.
🧭
Why keep the text too?

The embedding is for finding the chunk; the stored text is what you actually feed the LLM. You don't re-derive text from a vector — embeddings aren't reversible — so the original text rides along.

End-to-end example — from a document to a stored, queryable vector

Let's follow one real document all the way into the store, then run a query against it.

1
Start with a loaded document
source: "rag_guide.pdf", page 3
text: "RAG retrieves relevant passages before generating an answer.
       This grounds the model and reduces hallucination."
2
Chunk it (recursive, ~80 chars)
chunk_0: "RAG retrieves relevant passages before generating an answer."
chunk_1: "This grounds the model and reduces hallucination."
3
Embed each chunk → a vector (Chapter 6 pipeline)
chunk_0 → embed() → [ 0.021, -0.118,  0.067, … ]   (384-d)
chunk_1 → embed() → [-0.044,  0.092, -0.010, … ]   (384-d)
4
Assemble one record per chunk and store it
{
  id:        "rag_guide_p3_c0",
  embedding: [0.021, -0.118, 0.067, …],
  text:      "RAG retrieves relevant passages before generating an answer.",
  metadata:  {"source": "rag_guide.pdf", "page": 3}
}
# …and the same shape for chunk_1. Both go into the ANN index (HNSW).
5
Query time — the question is embedded the SAME way, then compared
Q: "How does RAG avoid making things up?"
   → embed(Q) = [0.018, -0.101, 0.058, …]

cosine(Q, chunk_0) = 0.71
cosine(Q, chunk_1) = 0.86   ← higher (it's about reducing hallucination)

return top-1 → chunk_1's TEXT goes into the LLM prompt.

The one rule that makes this work

The same embedding model must embed both the stored chunks and the query — they have to live in the same vector space for cosine similarity to mean anything. Mixing models (e.g. store with OpenAI, query with BGE) produces garbage rankings.

Difference between a vector store and a vector database?

Delete the old vector by ID, then add the new one with the same ID. Adding without deleting leaves stale duplicates that can surface contradictory context during retrieval.

Part 6

Approximate Nearest Neighbour Search

Brute force can't scale to millions of vectors. ANN indexes (IVF, HNSW) trade ~1% accuracy for 100–1000× speed — the engine inside every vector database.

Chapter 21

Why ANN? The Brute-Force Problem

Mathematical explanation

Exact ("brute force") search compares the query to every vector. With n documents of dimension D, each comparison is O(D) and there are n of them:

Costbrute = O(n · D)  +  O(n log n) to sort
Linear in both corpus size and dimensionality — fine for thousands, fatal for millions.
Worked example 
10,000,000 docs × 1536 dims = 15.36 billion multiply-adds per query
On a CPU (~3 GFLOP/s effective for this): ≈ 5–50 seconds per query 😱
→ unusable for interactive search
💡
The ANN bargain

Approximate Nearest Neighbour search gives up a tiny bit of recall (you might miss the true #1 result ~1% of the time) in exchange for 100–1000× speedup — turning 5 seconds into 5 milliseconds. Two dominant families: IVF (clustering) and HNSW (graph).

Chapter 22

IVF — Inverted File Index (Clustering)

Intuition

Group similar vectors into clusters once (offline). At query time, only search the few clusters nearest the query — skip the rest. Like searching a few aisles of a supermarket instead of every shelf in the store.

Algorithm
Offline (build):
  • Run k-means over all vectors → k clusters, each with a centroid.
  • Store each centroid → list of its member vector IDs (the "inverted list").
Online (query):
  • Compare query to all k centroids (cheap).
  • Pick the nprobe nearest clusters.
  • Brute-force search only within those clusters; return global top-k.
Step 1 compares Q to the 3 centroids (cheap). Step 2 searches only the nearest cluster's members. The blue clusters are never opened.
Worked example A — which cluster gets searched (the distance math)
1
After k-means we have 3 centroids (2-D for illustration)
C1 = (1.5, 7.0)   C2 = (5.0, 2.0)   C3 = (8.5, 6.5)
Query Q = (4.6, 2.9)
2
Compute Euclidean distance Q → each centroid
d(Q,C1) = √((4.6−1.5)² + (2.9−7.0)²) = √(9.61+16.81) = √26.42 = 5.14
d(Q,C2) = √((4.6−5.0)² + (2.9−2.0)²) = √(0.16 +0.81 ) = √0.97  = 0.98  ← smallest
d(Q,C3) = √((4.6−8.5)² + (2.9−6.5)²) = √(15.21+12.96) = √28.17 = 5.31
3
Pick nprobe nearest clusters, search only those
nprobe=1 → search C2 only.   (nprobe=2 → C2 then C1, for safety near boundaries)

Significance of nprobe: it's the recall⇄speed dial. nprobe=1 is fastest but may miss a true neighbour sitting just across a cluster border; raising it searches more clusters → higher recall, more cost.

Worked example B — the speedup, with numbers
1
Setup
N = 1,000,000 docs,  k = 1000 clusters  →  ~1000 docs per cluster,  nprobe = 5
2
Count the comparisons
Step 1 — Q vs all k centroids:        1000 comparisons
Step 2 — search nprobe clusters:      5 × 1000 = 5000 comparisons
Total ≈ 6,000   vs brute force 1,000,000   →  ~167× faster ✅
3
Rule of thumb
Optimal k ≈ √N.   For N=1M → k≈1000.   Cost ≈ k + nprobe·(N/k) comparisons.

Significance: choosing k≈√N balances the two terms (centroid scan vs in-cluster scan) to minimize total work.

✅ Pros

  • Huge speedup, simple idea
  • Tunable recall via nprobe
  • Memory-efficient

⚠️ Cons

  • Needs training (k-means) up front
  • Clusters unbalance as data grows → re-index
  • Boundary vectors can be missed

HNSW — Hierarchical Navigable Small World

Intuition

Build a multi-layer graph of vectors. Top layers are sparse with long-range links; bottom layers are dense with local links. Search zooms from coarse to fine — like GPS going world → country → city → street. Finds a near neighbour in ~log(N) hops.

What the pieces mean
  • Node = one vector.  Edge = "these two are near neighbours".  Layer = zoom level.
  • Six degrees of separation: sparse upper layers let any node reach any other in ~log(N) hops.
  • M = max neighbours per node (≈16).  efConstruction = build-time beam width.  efSearch = query-time beam width (higher = more accurate, slower).
Red path = the actual search: long hop A→G at the top, descend and refine G→E, descend again E→D. Only ~3 hops touch the data; the green nodes are the returned neighbours.

Worked example (1D): find the neighbour of query = 8

Vectors (1D): A=1  B=3  C=5  D=7  E=9  F=11  G=13

Layer 2:  A ──────── G
Layer 1:  A ── C ── E ── G
Layer 0:  A─B─C─D─E─F─G

Search query=8:
 L2: start A(1). |8−1|=7 vs |8−13|=5 → move to G(13)
 L1: at G(13). neighbour E(9): |8−13|=5 vs |8−9|=1 → move to E(9)
 L0: at E(9). neighbours D(7),F(11): |8−9|=1, |8−7|=1, |8−11|=3
     → D(7) and E(9) tie as nearest → return them
Hops: 3   vs   7 brute-force scans  ✅

The maths behind HNSW (paper notation)

HNSW (Malkov & Yashunin, 2016) borrows two ideas: skip-lists (probabilistic layering) and navigable small-world graphs (greedy routing). Two formulas matter.

① Layer assignment:   l = ⌊ −ln(U) · mL ⌋ ,   U ~ Uniform(0,1) ,   mL = 1/ln(M)
When a node is inserted, this draws its top layer. The −ln(U) term makes high layers exponentially rare, so layer 0 holds all N nodes, layer 1 holds ~N/M, layer 2 ~N/M², …
Significance: this exponential thinning is why search is O(log N) — the top layers are a tiny "express map" that lets you cross the dataset in a few long hops before refining locally.
② Search cost:   O( log N )  hops  (vs O(N) brute force)
Significance: at each layer greedy search moves to the neighbour closest to the query until no neighbour is closer, then drops a layer. Few layers × few neighbours each ⇒ logarithmic.

Layer-count example (why upper layers are sparse)

1
Pick M (max neighbours per node)
M = 16  →  m_L = 1/ln(16) = 1/2.77 = 0.361
2
Expected nodes per layer for N = 1,000,000
Layer 0: N            = 1,000,000   (everything)
Layer 1: N/M          ≈    62,500
Layer 2: N/M²         ≈     3,906
Layer 3: N/M³         ≈       244
Layer 4: N/M⁴         ≈        15
Top layer ≈ log_M(N) = ln(1e6)/ln(16) ≈ 5 layers
3
Takeaway

A million-vector graph is only ~5 layers deep, and each greedy step examines ≤ M neighbours — so a query touches on the order of M · logM(N) ≈ 16·5 = 80 nodes, not 1,000,000.

Interview questions
Why is brute-force search impractical at scale?
It's O(n·D) per query — every vector compared on every dimension. At 10M×1536 that's ~15B operations per query (seconds), far too slow for interactive use.
IVF vs HNSW — when to pick which?
IVF: lower memory, needs k-means training, re-index as data grows — good for huge static datasets. HNSW: log(N) queries, native incremental inserts, higher memory — the default for most dynamic vector DBs.
What do efSearch / nprobe control?
Both trade recall for speed at query time. Higher efSearch (HNSW) explores more of the graph; higher nprobe (IVF) searches more clusters — more accurate, slower.

Retrieval

The retriever is the policy layer over the index — deciding what, how many, and how diverse the chunks handed to the LLM are. From plain similarity to MMR, hybrid, and self-querying.

Chapter 24

Retriever Architecture

Concept

A retriever wraps a vector store and exposes one method: invoke(query) → docs. The vector store does raw similarity math; the retriever adds strategy — k, diversity, thresholds, filters, fusion, re-ranking. In LangChain everything is a retriever, so they compose.

The store finds candidates; the retriever decides which and how many actually reach the LLM.

Concept

The redundancy problem

Pure similarity often returns 4 near-duplicate chunks all making the same point. The LLM then sees one idea four times and misses other relevant facts — which is exactly what MMR fixes.

Chapter 26

MMR — Maximum Marginal Relevance

Concept

MMR balances relevance to the query against diversity among the chosen docs, so you get coverage instead of duplicates. It greedily builds the result set one doc at a time.

Mathematical explanation
MMR(dᵢ) = λ · sim(dᵢ, q) − (1 − λ) · maxdⱼ∈S sim(dᵢ, dⱼ)
sim(dᵢ,q) = relevance to query · max sim(dᵢ,dⱼ) = similarity to already-picked docs (redundancy) · λ = lambda_mult (1 = pure relevance, 0 = pure diversity).
Significance of each term: the first term rewards a candidate for being close to the query; the second term penalizes it for being close to something already chosen. Subtracting the second is what kills duplicates — a candidate near an already-picked doc gets docked, so the next pick brings new information. λ sets how aggressively you trade relevance for coverage. Introduced by Carbonell & Goldstein (1998).
Worked example (λ = 0.5) 
Candidates (similarity to query q):
  C1 SGD optimizes nets        0.91
  C3 Adam adapts LR            0.85
  C4 Overfitting w/ small data 0.78

Round 1 (S empty): MMR = 0.5·sim(·,q).  C1 highest (0.455) → pick C1.  S={C1}
Round 2:
  C3: 0.5·0.85 − 0.5·sim(C3,C1)=0.425 − 0.5·0.40 = 0.425−0.200 = 0.225
  C4: 0.5·0.78 − 0.5·sim(C4,C1)=0.390 − 0.5·0.30 = 0.390−0.150 = 0.240 ← higher
  → pick C4 (less redundant), even though C3 was more similar to q.  S={C1,C4}

Result: C1 (SGD) + C4 (overfitting) — two DIFFERENT ideas, not two SGD chunks ✅

Chapter 29

Hybrid Search (BM25 + Dense + RRF)

Concept

BM25 nails exact keywords; dense embeddings catch meaning. Hybrid search runs both and fuses the ranked lists — usually with Reciprocal Rank Fusion (RRF).

Two retrievers, one fused ranking. Docs that rank high in both lists win.
Mathematical explanation — RRF
RRF(d) = Σrankers  1 / (k + rank(d))   (k = 60 by convention)
Each list contributes by a doc's rank (not raw score), so incomparable score scales don't matter.
Worked example 
BM25 ranks:  1.D7  2.D1  3.D3        Dense ranks: 1.D5  2.D7  3.D2   (k=60)
RRF(D7) = 1/(60+1) + 1/(60+2) = 0.01639 + 0.01613 = 0.03252 ← highest
RRF(D5) = 0        + 1/(60+1) =                       0.01639
RRF(D1) = 1/(60+2) + 0        =                       0.01613
→ D7 wins: it ranked high in BOTH lists (cross-list consensus) ✅

What problem does MMR solve and how?
Redundant top-k results. MMR greedily picks docs maximizing λ·relevance − (1−λ)·max-similarity-to-already-picked, so each new doc adds new information instead of repeating prior ones.

Part 8

Re-Ranking

A two-stage pattern: retrieve many candidates fast, then re-score the top-N with a slow-but-accurate cross-encoder for precision where it counts.

Chapter 31

Why Re-Rank? Two-Stage Retrieval

Concept

First-stage retrieval optimizes for speed (compare pre-computed embeddings). It's approximate. A second stage re-scores just the top candidates with a model that reads query and document together — far more accurate, but too slow to run over the whole corpus.

Cast a wide net cheaply, then precisely rank the keepers. Best precision-per-dollar.
Chapter 32

Cross-Encoder vs Bi-Encoder

Concept

A bi-encoder embeds query and doc separately, then compares vectors — fast, pre-computable (this is normal retrieval). A cross-encoder feeds [query, doc] through one transformer together, so every query token attends to every doc token — far more accurate, but must run per pair at query time.

Bi-encoder = first-stage retrieval (scalable). Cross-encoder = second-stage re-rank (precise).

Why the cross-encoder is more accurate — the mechanism

A bi-encoder must compress a whole document into one fixed vector before it ever sees the query. Information the query cares about may have been averaged away. A cross-encoder processes query and document jointly, so self-attention lets every query token look at every document token.

Bi-encoder:  score = cos( f(q), f(d) )   — two independent encodings, compared after
Cross-encoder:  score = g( [CLS] q [SEP] d [SEP] )  — one joint encoding → scalar
Significance: in the bi-encoder, f(q) and f(d) are computed in isolation, so the doc vector is query-agnostic (but precomputable → scales to millions). The cross-encoder's g(·) sees the pair together, modelling token-level interactions (e.g. that the query's "capital" lines up with the doc's "Paris") — much sharper, but it must run once per (query, doc) pair, so it can't scan the corpus.

RAG Fusion (Multi-Query + RRF)

Generate several sub-queries from the user's question, retrieve for each, then fuse all lists with RRF (Ch 29). Surfaces consensus documents and boosts recall on ambiguous queries.

More phrasings → broader recall; RRF rewards docs that multiple sub-queries agree on. Cost: several extra retrievals.
The fusion maths — Reciprocal Rank Fusion (RRF)
RRF(d) = Σi  1 / ( k + ranki(d) )   k = 60 (standard)
ranki(d) = position of doc d in result list i (1 = top). Sum over all sub-query lists.
Significance: RRF fuses by rank, not score, so it doesn't matter that different sub-queries return incomparable similarity scales. The 1/(k+rank) shape gives big credit for being near the top of any list and rewards docs that appear in many lists — i.e. consensus. The constant k=60 softens the gap between rank 1 and rank 2 so a single list can't dominate.

Chapter 35

HyDE — Hypothetical Document Embeddings

The problem it solves

Dense retrieval needs a good query encoder trained on (query, relevant-doc) pairs. For a new domain or language you often have no such training data (zero-shot). HyDE (Gao et al. 2022) sidesteps this: it never tries to encode the question well — instead it has an LLM imagine an answer and encodes that with an unsupervised dense encoder.

The exact pipeline from the paper

HyDE has two models working together — a generative LLM (the paper uses InstructGPT) and an unsupervised contrastive encoder (the paper uses Contriever). The flow:

HyDE pipeline (Gao et al. 2022, Fig. 1 redrawn): the LLM generates a fake answer, the unsupervised Contriever encodes it, and the resulting vector retrieves real passages. The fake doc is discarded after encoding.
What is Contriever, and why HyDE needs it

Contriever (Izacard et al. 2021) is a BERT-based dense encoder trained without any labels, using contrastive learning: two random crops of the same document are a positive pair (should be close); crops from different documents are negatives (should be far). Its training objective is InfoNCE:

L = − log [ exp(s(q,k⁺)/τ) / ( exp(s(q,k⁺)/τ) + Σi exp(s(q,kᵢ⁻)/τ) ) ]
s(a,b)= dot product of encodings · k⁺ = positive (same doc) · kᵢ⁻ = negatives · τ = temperature.
Why HyDE specifically uses it: Contriever is unsupervised, so it embeds any text — question or passage — into one sensible space without needing (query,doc) training pairs. HyDE's whole selling point is "zero-shot": no relevance labels. A supervised query encoder would defeat that purpose, so HyDE pairs the LLM generator with the label-free Contriever.
The retrieval maths (with the multi-sample average)

The paper doesn't generate just one hypothetical doc — it samples N of them and averages their encodings (plus the query encoding) to reduce the effect of any single bad generation:

v̂ = (1 / (N+1)) · [ f(q) + Σk=1..N f(hk) ]
retrieve:   doc* = argmaxd   v̂ · f(d)
Significance of averaging: each generated hk is a noisy "guess" of where the answer lives. Averaging N of them (the paper uses N=8) cancels idiosyncratic errors and keeps the shared signal — the topic. Adding f(q) keeps a foot in the literal query in case generation drifts.
Worked numeric example
1
Query & two sampled hypothetical docs, encoded (toy 3-D, then normalized)
f(q)  = [0.20, 0.10, 0.00]      (literal question — short, vague)
f(h₁) = [0.60, 0.50, 0.10]      (generated answer #1)
f(h₂) = [0.50, 0.55, 0.05]      (generated answer #2)
2
Average (N=2 → divide by 3)
v̂ = ([0.20,0.10,0.00]+[0.60,0.50,0.10]+[0.50,0.55,0.05]) / 3
   = [1.30, 1.15, 0.15] / 3
   = [0.433, 0.383, 0.050]
3
Score two real corpus docs by dot product with v̂
d_relevant   = [0.55, 0.50, 0.05]   v̂·d = 0.433·0.55 + 0.383·0.50 + 0.05·0.05 = 0.4325
d_offtopic   = [0.05, 0.05, 0.80]   v̂·d = 0.433·0.05 + 0.383·0.05 + 0.05·0.80 = 0.0808
4
Compare with embedding the bare query instead
f(q)·d_relevant = 0.20·0.55 + 0.10·0.50 + 0 = 0.160
f(q)·d_offtopic = 0.20·0.05 + 0.10·0.05 + 0 = 0.015
gap (query only) = 0.160 − 0.015 = 0.145
gap (HyDE)       = 0.433 − 0.081 = 0.352   → HyDE separates relevant from off-topic ~2.4× better
💡
The key insight

The hypothetical doc can contain factual errors and HyDE still works — because it's used only as a retrieval probe, never shown to the user. The Contriever's dense bottleneck keeps the topic and discards most hallucinated specifics, so the probe lands among real answer passages. The final answer is generated from those real passages.

Trade-off: +1 (or +N) LLM generation calls per query; shines in zero-shot / new-domain settings where you can't train a query encoder.

CRAG — Corrective Retrieval-Augmented Generation

The problem it solves

Plain RAG always feeds retrieved docs to the generator — even when retrieval returns garbage. CRAG (Yan et al. 2024) inserts a retrieval evaluator that judges document quality first, then applies one of three corrective actions before generation.

The full architecture (paper, Fig. 2 redrawn)
CRAG (Yan et al. 2024): a fine-tuned T5 evaluator scores retrieval, routing to knowledge refinement (good), web search (bad), or both (ambiguous). Note the "decompose-then-recompose" strip filtering — that's the contriver/refiner you flagged as missing.
① The retrieval evaluator — the maths

A lightweight T5-large is fine-tuned to output a relevance confidence score for each retrieved document given the query. The per-query confidence is aggregated over the k docs, then thresholded into three buckets:

conf(q) = aggregatei=1..k  Eval(q, dᵢ) ,   Eval(q,dᵢ) ∈ [−1, 1]
action =  CORRECT  if conf ≥ τhi  ·  INCORRECT if conf ≤ τlo  ·  AMBIGUOUS otherwise
Significance: the two thresholds τhi, τlo turn a continuous relevance score into a decision. Tighten τhi → CRAG distrusts its own corpus and hits the web more (fresher but slower, costlier); loosen it → behaves more like plain RAG.
② Knowledge refinement — the "decompose-then-recompose" algorithm

This is the refiner that was missing before. Even a relevant document is mostly filler around a few useful sentences. CRAG splits it into knowledge strips, scores each, and keeps only the relevant ones:

1
Decompose the retrieved doc into strips (≈ 1–2 sentences each)
doc → [strip₁, strip₂, strip₃, strip₄]
2
Score each strip with the same evaluator, filter by a strip threshold
strip₁  score 0.81  ✔ keep
strip₂  score 0.12  ✘ drop (filler)
strip₃  score 0.74  ✔ keep
strip₄  score −0.40 ✘ drop (off-topic)
3
Recompose the surviving strips into clean context
refined = strip₁ ⊕ strip₃   → dense, on-topic context for the generator

Significance: this is internal "noise removal" — it raises context precision (Ch 42) so the generator isn't distracted by irrelevant sentences inside an otherwise-relevant document.

③ Worked end-to-end example
1
Query & evaluator scores
q = "What is the 2024 Nobel Prize in Physics for?"
retrieved corpus docs (all from a 2022 snapshot):
  d₁ score −0.6 , d₂ score −0.8 , d₃ score −0.5
conf = max = −0.6     τ_hi=0.6, τ_lo=−0.2  →  conf ≤ τ_lo  →  INCORRECT
2
Corrective action → web search with query rewriting
LLM rewrites q → keywords "2024 Nobel Prize Physics winners"
web search → fresh pages → refine into strips → keep the 2 relevant strips
3
Generate from the corrected (web) knowledge
answer grounded in fresh web strips, NOT the stale 2022 corpus ✅
💡
CRAG vs plain RAG vs Self-RAG

Plain RAG trusts retrieval blindly. CRAG adds an external, plug-and-play evaluator + refiner + web fallback — it works with any frozen LLM. Self-RAG (next chapter) instead trains the generator itself to self-critique. CRAG = bolt-on corrector; Self-RAG = built-in corrector.

Self-RAG (Reflection Tokens)

The model itself decides when to retrieve and critiques its own output using special reflection tokens — [Retrieve], [Relevant], [Supported], [Useful] — retrieving only when needed and checking that claims are grounded.

Query → [Retrieve?] yes → fetch docs
      → [Relevant?] Doc1 yes, Doc2 no  → keep Doc1
      → generate → [Supported?] yes → [Useful?] yes → answer ✅
The reflection tokens (what the model is trained to emit)
TokenAsksEffect
RetrieveDo I need to retrieve here?{yes / no / continue} — skips retrieval when not needed
IsRelIs this passage relevant?{relevant / irrelevant} — filters retrieved docs
IsSupIs my statement supported by it?{fully / partially / no} — groundedness check
IsUseIs the answer actually useful?{1…5} — final quality score
💡
Significance — vs CRAG

CRAG bolts an external evaluator onto a frozen LLM. Self-RAG (Asai et al. 2023) bakes the self-critique into the model itself: it's fine-tuned to emit these special tokens, so it learns when to retrieve and to verify its own claims against the evidence as part of generation. More powerful, but requires training the model — you can't just prompt it.

Worked inference trace (image form)

At inference Self-RAG processes the answer segment by segment. For a factual query it emits Retrieve=yes, pulls k passages in parallel, generates a candidate continuation for each, and scores them with its own critique tokens — then keeps the best:

Self-RAG (Asai et al. 2023) generates a candidate per passage, critiques each with reflection tokens, and keeps the highest-scoring segment. The irrelevant Nobel passage is rejected by ⟨IsRel⟩.
The segment-selection maths
score(d) = log p(segment | q, d)  +  Σt∈{IsRel,IsSup,IsUse} wt · st(d)
p(segment|q,d) = the LM's own likelihood of the continuation · st = critic-token probabilities (e.g. P(IsSup=fully)) · wt = tunable weights.
Significance: at inference you can turn the dials wt with no retraining — raise wIsSup to prioritize groundedness (fewer hallucinations), raise wIsUse to prioritize helpfulness. The critique is part of the score, so the model literally selects the most grounded continuation rather than the merely most fluent one.

Trade-off: needs special training (the critic + generator are fine-tuned to emit/consume reflection tokens); more precise and avoids needless retrieval, but slower inference.

Graph RAG (Knowledge Graphs)

Store knowledge as a graph of entities (nodes) and relationships (edges). Traverse it for multi-hop relational questions that flat vector search struggles with.

"Where did Apple's CEO work before?" is a 2-hop traversal (Apple→CEO→Tim Cook→WORKED_AT→IBM) — natural in a graph, hard for flat retrieval.
# Cypher (Neo4j) · docs: https://python.langchain.com/docs/integrations/graphs/neo4j_cypher/
MATCH (a:Company {name:"Apple"})-[:ACQUIRED]->(c:Company)
RETURN c.name

How Graph RAG works — two pipelines

① Ingestion pipeline (build the graph)
  • Chunk the documents as usual
  • LLM extracts entities (nodes) and relationships (edges) from each chunk
  • Write nodes + edges into a graph DB (e.g. Neo4j)
  • Optionally also embed chunks for hybrid graph + vector retrieval
② Retrieval pipeline (answer a query)
  • Identify entities mentioned in the query
  • Traverse the graph from those entities (multi-hop)
  • Collect connected nodes/edges as context
  • LLM generates the answer from that subgraph

Microsoft GraphRAG — community summaries for "global" questions

Plain graph traversal answers local questions ("who did X acquire?"). But questions like "what are the main themes across this whole corpus?" can't be answered by any single subgraph. Microsoft's GraphRAG (Edge et al. 2024) adds a clustering + summarization layer:

Microsoft GraphRAG (Edge et al. 2024): cluster the graph into communities with the Leiden algorithm, summarize each with an LLM into hierarchical "community reports", then route local (entity) vs global (themes, map-reduce over reports) queries differently.
💡
Why community summaries matter

A "global" question ("summarize the key risks across 10,000 reports") has no single passage that answers it — vector RAG retrieves k chunks and misses the forest for the trees. GraphRAG's map-reduce over community summaries lets the model reason over the whole corpus structure, which flat retrieval fundamentally cannot.

⚠️
Limitations

Graph RAG is powerful but costly: graph construction is expensive (LLM entity/relation extraction plus community summarization over the whole corpus), extraction quality is imperfect (missed or wrong relationships propagate), it's harder to update than a vector store, and it shines mainly on genuinely relational or global questions — overkill for simple lookup.

Multi-Modal RAG

Retrieve over text + images. Either embed images with CLIP into a shared text-image space, or caption images with a vision LLM and embed the captions as text.

Intuition — CLIP contrastive space

CLIP (Radford et al. 2021) trains a text encoder and an image encoder so that matching (image, caption) pairs land at nearly the same point in one shared space, while mismatched pairs are pushed apart. That shared space is what lets you search images with text and vice-versa.

Two encoders, one space. A text query and the image that matches it end up neighbours, so cosine similarity works across modalities.
The contrastive objective (the maths)

For a batch of N (image, text) pairs, CLIP computes all N×N cosine similarities and trains so the N correct pairs (the diagonal) score highest. It's a symmetric cross-entropy over rows and columns (InfoNCE):

L = ½ [ CE(softmax(S/τ), I) over rows  +  CE(softmax(Sᵀ/τ), I) over columns ]
Sij = cos(imgᵢ, textⱼ) = N×N similarity matrix · I = identity (correct pairs on the diagonal) · τ = temperature.
Significance: the loss pulls each image toward its caption and pushes it away from the other N−1 captions in the batch (and symmetrically for text). Maximizing the diagonal of S = aligning matching pairs; minimizing off-diagonal = separating mismatches. The temperature τ sharpens the softmax (small τ = harsher push/pull). This is what builds the shared space.
Worked example — text-to-image retrieval with CLIP
1
Encode the text query and candidate images into the same space
query "a dog playing fetch" → text encoder → t = [0.21, 0.39, …]
img_A (dog photo)           → image encoder → a = [0.23, 0.41, …]
img_B (cat photo)           → image encoder → b = [0.88,-0.12, …]
2
Score by cosine similarity (same metric as text RAG!)
cos(t, a) = 0.92   ← dog image, high
cos(t, b) = 0.10   ← cat image, low
3
Return the nearest image(s)
top-1 = img_A. The text query retrieved an IMAGE — no captions needed.

The RAGAS evaluation dataset

Every RAGAS metric is computed over a dataset of test rows. Each row carries up to five fields — knowing which fields a metric uses tells you what it can measure.

FieldMeaning
user_inputThe question asked

Each metric, precisely

① Context Recall — completeness of retrieval

Did the retriever fetch all the information needed to answer? It checks for completeness — how many of the relevant pieces were successfully retrieved. Focus: not missing anything important.

Context Recall = (ground-truth claims supported by retrieved context) / (total ground-truth claims)
Significance: this is the ceiling on your whole system. If the retriever didn't fetch the fact, no LLM can answer correctly from context — low recall means "fix retrieval first".
Worked example — reference answer has 4 key facts; the retrieved chunks cover 3 of them. 
fact 1 (birth date)      ✔ present in retrieved context
fact 2 (relativity)      ✔ present
fact 3 (Nobel 1921)      ✔ present
fact 4 (born in Ulm)     ✘ missing from retrieved context

Context Recall = 3 / 4 = 0.75

② Context Precision — ranking quality

Not just whether the right chunks were retrieved, but where they appear in the list — are relevant chunks ranked above irrelevant ones? Computed as the mean of precision@k across the retrieved context.

Context Precision = ( Σk precision@k · relk ) / (total relevant) ,   precision@k = (relevant in top-k) / k
Significance: retrieval that buries the right chunk at rank 8 wastes the LLM's limited context budget on noise first. High context precision means the good stuff is at the top.
Worked example — 5 retrieved chunks, ranking matters (✔ = relevant): 
rank 1  ✔ relevant   precision@1 = 1/1 = 1.00
rank 2  ✘ irrelevant precision@2 = 1/2 = 0.50   (not counted: rel=0)
rank 3  ✔ relevant   precision@3 = 2/3 = 0.67
rank 4  ✘            —
rank 5  ✘            —

Context Precision = (1.00 + 0.67) / 2 relevant = 0.83
→ would be 1.0 if both relevant chunks were at ranks 1 & 2.

③ Faithfulness — the hallucination detector

Can every claim in the response be traced back to the retrieved context? A faithful answer says only what the chunks support — no outside knowledge, no invented facts.

Faithfulness = (claims supported by context) / (total claims in response)
Worked example — query "Tell me about Albert Einstein." 
Response claims:
  1. Born 14 March 1879 in Germany        ✔ supported by context
  2. Developed the theory of relativity   ✔ supported
  3. Won the Nobel Prize in Physics 1921  ✔ supported
  4. Known for work on quantum mechanics  ✔ supported
  5. He had an IQ of 160                   ✘ NOT in context (invented)

Faithfulness = 4 / 5 = 0.80   → the fabricated IQ claim drags the score down

④ Response Relevancy — on-topic-ness

Does the answer actually address the question asked? This checks relevance, not factual correctness — an answer can be faithful yet wander off-topic, or be on-topic yet unfaithful.

Answer Relevancy = mean  cos( embed(qᵢ′), embed(original q) )
RAGAS asks an LLM to generate n questions qᵢ′ that the answer would perfectly address, embeds them, and compares to the real question.
Significance: if the answer is on-topic, the questions reverse-generated from it look like the real question (high cosine). A rambling or evasive answer produces unrelated reverse-questions → low score.
Worked example 
Original q: "What is the capital of France?"
Answer: "Paris is the capital of France."
Reverse-generated questions from the answer:
  q1' "What is France's capital?"        cos=0.99
  q2' "Which city is the French capital?" cos=0.96
  q3' "What is the capital city of France?" cos=0.98
Answer Relevancy = mean(0.99,0.96,0.98) = 0.98  (highly on-topic)

⑤ Noise Sensitivity — robustness (lower is better)

How much does the LLM get misled by irrelevant chunks in the context? It measures whether the model stays faithful to the relevant chunks or gets distracted by noise and starts making incorrect claims. A robust system has low noise sensitivity.

Noise Sensitivity = (incorrect claims attributable to irrelevant/noisy chunks) / (total claims)
Significance: unlike the others, lower is better. It isolates the failure mode where adding a distracting chunk causes the model to assert something wrong — a key robustness test before you trust hybrid/large-k retrieval that inevitably pulls in some noise.
Worked example 
Context = [3 relevant chunks + 1 noisy chunk about a different person].
Answer makes 5 claims; 1 of them is wrong and traces to the noisy chunk.
Noise Sensitivity = 1 / 5 = 0.20   (lower = more robust)
🧭
Faithfulness vs Response Relevancy — don't confuse them

Faithfulness asks "is it grounded in the context?" (hallucination). Response Relevancy asks "does it answer the question?" (on-topic). You want both high — a confident, well-grounded answer to a different question is still useless.

These compose — production stacks layer them

The techniques aren't mutually exclusive. A typical strong pipeline is hybrid retrieve → re-rank → parent-document expand → generate, then add CRAG/Self-RAG for reliability, and wrap the whole thing in an agentic router once you have multiple sources.

⚠️
The most common mistake

Throwing vector RAG at a numeric/aggregation question ("total revenue by region"). Embeddings find similar text, not computed answers — they can't SUM or GROUP BY. Route those to Text-to-SQL. Likewise "what's the current X?" needs a live API call, not a stale index.

The three patterns underneath all of them

Pick the pattern by the shape of the answer's source, then layer access-control (metadata), freshness (caching/time filters), and reliability (CRAG/Self-RAG/guardrails) on top.
🧭
At enterprise scale it's all of the above

A "company-wide assistant" isn't one RAG — it's an agentic router that classifies each question and dispatches it to the right sub-pipeline (Document / SQL / Graph / API), then fuses the results. Build the individual pipelines first; add the router last.

Part 12 summary — Don't pick a RAG by its data label — pick by the shape of the answer: unstructured text → document RAG (hybrid + re-rank + parent-doc); structured/live → Text-to-SQL / API; relational or non-text → Graph / Multimodal. Layer access control, freshness, and reliability on top, and at scale let an agentic router choose per question. Start simple; add the cheap high-ROI upgrades (re-ranking, parent-document) first.
